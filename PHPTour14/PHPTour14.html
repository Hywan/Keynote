Définitions

System Under Test (SUT) : système qui sera exécuté

  • préambule (given)
  • données (when)
  • oracle (then)
  • cas de test
  • place le SUT dans un état particulier
  • test, exécute le SUT
  • établit le verdict, est-ce que le produit de l’exécution et l’état du SUT après son exécution sont ceux attendus ou non ?

Valeurs du verdict : succès, échec ou inconclusif

Suite de tests : ensemble de cas de tests

Nature des tests

Nous : test unitaire, boîte noire et fonctionnel 📖

Design by Contract

Contrat :

  • inventé par B. Meyer en 1992 dans le langage Eiffel 📖 📖
  • décrit un modèle en utilisant des annotations
  • exprime des contraintes formelles : préconditions, postconditions et invariants
  • caller : « je m'engage à satisfaire ta précondition »
  • callee : « je m'engage à établir ma postcondition »
  • invariants satisfaits avant et après l'exécution du callee

Langages de contrats

Chaque « grand » langage de programmation à un langage de contrat :

  • Java Modeling Language (JML) pour Java 📖
  • ANSI/C Specification Language (ACSL) pour C 📖
  • Spec# pour C# 📖
  • JSConTest pour Javascript 📖

Pour PHP ? Rien !

Contract-based Testing

Inventé par B. K. Aichernig en 2003 📖

Principe :

  • utilise les préconditions pour générer des données de tests
  • utilise les postconditions et invariants pour établir le verdict du test à l'exécution, grâce à un Runtime Assertion Checker (RAC)

Problèmes :

  • contraintes souvent exprimées avec des formules logiques
  • difficile d'en générer des données

Solutions

Faire du Contract-based Testing en PHP en 3 étapes

  1. un (nouveau) langage de contrats : 📖
    • adapter à PHP
    • pragmatique
    • facile à utiliser pour le développeur
    • assemblant plusieurs méthodes de test
  2. des générateurs de données : 📖 📖
    • pour tous les types usuels
  3. générer automatiquement des suites de tests unitaires

1 Praspel et les domaines réalistes

Domaines réalistes

Naïvement, les domaines réalistes :

  • raffinent les types usuels, comme les entiers, les chaînes de caractères, les tableaux etc.
  • permettent d'exprimer des données plus complexes, comme des grammaires, des graphes etc.

Réaliste signifie qu'ils sont conçus pour spécifier des domaines de données pertinents pour un contexte spécifique

Exemple : adresse email

Caractéristiques

Conçus pour répondre aux problématiques du test

Ils ont les deux caractéristiques suivantes :

  • la prédicabilité permet de valider qu'une donnée appartient à l'ensemble des valeurs décrites par le domaine réaliste
  • la générabilité permet de générer une valeur décrite par le domaine réaliste

Très important !

Hiérarchie

Une classification possible : verticale

Héritage permet de raffiner les caractéristiques :

  • Color hérite de String
  • Boundinteger hérite de Integer
  • tous sont enfants de la classe Realdom

Univers Urealdom :

Exemple

class        Boundinteger
  extends    Integer
  implements IRealdom\Interval,
             IRealdom\Finite,
             IRealdom\Nonconvex,
             IRealdom\Enumerable {

    protected $_arguments = [
        'Constinteger lower' => PHP_INT_MIN,
        'Constinteger upper' => PHP_INT_MAX
    ];

    public function predicate ( $q ) {

        return    parent::predicate($q)
               && $q >= $this['lower']->getConstantValue()
               && $q <= $this['upper']->getConstantValue();
    }

    public function sample ( $sampler ) {

        return $sampler->getInteger(
            $this['lower']->sample($sampler),
            $this['upper']->sample($sampler)
        );
    }

    // ...
}

Praspel

PHP Realistic Annotation and SPEcification Language :

  • un langage et un framework de test à partir de contrats
  • un langage d'annotation (formellement un Behavioral Interface Specification Language📖), écrit dans les commentaires /** … */
  • permet d'exprimer des préconditions, postconditions et invariants
  • inspiré de JML et ACSL, mais inspiré seulement
  • utilise intensivement les domaines réalistes

Vue générale

class C {

    /**
     * @invariant I;
     */
    protected $a;

    /**
     * @requires P;
     * @behavior α {
     *     @description 'Apply the α process.';
     *     @requires P_α;
     *     @behavior β {
     *         @requires  P_α_β;
     *         @ensures   Q_α_β;
     *         @throwable T_α_β;
     *     }
     * }
     * @behavior γ {
     *     @requires P_γ;
     *     …
     * }
     * @default {
     *     @description 'Apply the fallback process.';
     *     @ensures   Q_D;
     *     @throwable T_D;
     * }
     */
     public function f ( ) { }
}

@invariant exprime un invariant

@requires exprime une précondition

@ensures exprime une postcondition normale

@throwable exprime une postcondition exceptionnelle ; de la forme T_C with T_E

@behavior exprime un comportement

Les @behavior peuvent s'imbriquer

@default exprime un comportement par défaut, avec une clause @requires implicite

@description décrit informellement un comportement

Expressions

P, Q et T_E représente une expression :

  • une déclaration
  • un prédicat
  • une qualification

Déclaration :

  • associe un ou plusieurs domaines réalistes à une variable du SUT (arguments) ou du modèle (let)
  • plusieurs = disjonction de domaines réalistes
  • \result représente le résultat du SUT
  • \old(i) représente la valeur de i dans le pré-état (avant l'exécution du SUT)

Prédicat :

  • toutes les constructions de Praspel supportent la validation et la génération mais…
  • … toutes les contraintes ne peuvent pas être exprimées
  • \pred(p) où p est du code PHP pur (un prédicat en forme normale conjonctive, CNF)

Qualification :

  • qualifier une donnée avec un adjectif
  • « mot-clé » représentant des contraintes usuelles
  • facilite la vie, facile à lire, évite les erreurs

Exemple

class Filesystem {

    const SIZE = 1024;

    /**
     * @invariant _usage: boundinteger(0, static::SIZE);
     */
    protected $_usage = 0;

    /**
     * @invariant _map: array([to class('File')], 0..0xffff);
     */
    protected $_map = array();

    /**
     * @ensures \result: this->_usage;
     */
    public function getUsage ( ) { /* … */ }

    /**
     * @requires file: class('File') and
     *           index: 0..0xffff or void;
     * @behavior full {
     *     @description 'The filesystem is full.';
     *     @requires  \pred('$this->getUsage() + $file->getSize()
     *                           > static::SIZE');
     *     @ensures   \none;
     *     @throwable AllocationException e with
     *                    file->isAttached(): false and
     *                    e->getFilesystem(): this;
     * }
     * @default {
     *     @ensures file->isAttached: true and
     *              \result: boolean();
     * }
     */
    public function store ( File $file, $index = null ) { /* … */ }
}

Évaluation du prédicat

Génération de tableaux

Dans PHP :

  • très utilisés
  • représentent toutes sortes de collections
  • associatifs

Dans Praspel :

  • la génération standard introduit trop de rejets
  • impossible d'exprimer des contraintes précises
  • utilisation de la construction \pred(p)

👉 Processus dédié à la génération de tableaux nécessaire

Approche

  1. étude des propriétés les plus populaires sur les tableaux (sur plus de 60 projets, plus 5 millions de LOC) ; retenues :

    • count
    • array_key_exists
    • in_array
  2. (syntaxe) extension de Praspel pour les inclure nativement
  3. (sémantique) transformation en propriétés ensemblistes
  4. solveur ensembliste

Syntaxe des propriétés

Déclaration de tableau : a: array(D, L)

Paire : a[K]: V

  • toute valeur dans K est la clé d'une paire dans le tableau a
  • toute paire dont la clé est dans K a sa valeur dans V

Exemple :

a[7..8]: true or 42

équivaut à :

   array_key_exists(7, $a)
&& array_key_exists(8, $a)
&& (in_array(true, $a) || in_array(42, $a))
&& ($a[7] === true     || $a[7] === 42)
&& ($a[8] === true     || $a[8] === 42)

Clé : a[K]: _

  • toute valeur de K doit être une clé du tableau a

Exemple :

a[7..8]: _

équivaut à :

   array_key_exists(7, $a)
&& array_key_exists(8, $a)

Valeur : a[_]: V

  • toute valeur de V doit être une valeur du tableau a

Exemple :

a[_]: 'foo'

équivaut à :

in_array('foo', $a)

Unicité : a is unique

  • toutes les clés sont uniques (intrinsèque au tableau)
  • toutes les valeurs sont uniques

Exemple :

a is unique

équivaut à :

// for simple cases
count($a) === count(array_unique($a, SORT_REGULAR))

Mais aussi…

a[K]!: V

  • toute valeur de K est la clé d'une paire dans le tableau a
  • toute paire dont la clé est dans K n'a pas sa valeur dans V

a[K]!: _

  • aucune paire du tableau a n'a sa clé dans K

a[_]!: V

  • …

Sémantique ensembliste

Un Constraint Satisfaction Problem et un triplet ( Σ, Δ, Π ) où…

Les variables σ ∈ Σ sont la taille S, les ensembles X et Y, la fonction totale H…

Cardinalité : card ( X ) = S , S ≥ 0 , card ( X ) ≥ card ( Y ) …

Domaines et co-domaines : Y = H ^ ( X ) , X = ∪ 1 ≤ i ≤ j X i …

Paires : K ⊆ X , H ^ ( K ) ⊆ V , H ^ ( K ) ∩ V = ∅ … 📖

Fonctionne en deux étapes :

  1. propagation et consistance
  2. labelling

Exemple :

a     : array([to string('a', 'e', 1)], length)
a[0]  : 'b' or 'c'
a is unique

a comme solution possible :

    0 => 'c',

Grammaires

En 1956, N. Chomsky établit une hiérarchie des langages formels📖 :

  1. grammaires générales (unrestricted grammars)
  2. grammaires contextuelles (context-sensitive grammars)
  3. grammaires algébriques (context-free grammars)
  4. grammaires régulières (regular grammars)

Exemples de…

  • … langages réguliers : expressions régulières POSIX ou PCRE
  • … langages algébriques : JavaCC, yacc, PP 📖 (nouveau !)

Langage PP

Informellement : grammaire = lexèmes + règles

Lexème (token) : plus petite unité lexicale composant un mot (donnée textuelle)

%token namespace_in:name value -> namespace_out

Exemple :

%token number \d+

Règle (rule) : décrit l'enchaînement des lexèmes les uns par rapport aux autres

  • rule() : pour invoquer une règle
  • <token> et ::token:: : pour déclarer un lexème
  • | : pour une disjonction (un choix)
  • (…) : pour grouper
  • e?, e+, e* et e{x,y} pour répéter
  • #node pour créer un nœud
  • token[i] pour unifier

Exemple simplifié avec JSON

Lexèmes :

%skip   space          \s
// Scalars.
%token  true           true
%token  false          false
%token  null           null
// Strings.
%token  quote_         "        -> string
%token  string:string  [^"]+
%token  string:_quote  "        -> default
// Objects.
%token  brace_         {
%token _brace          }
// Arrays.
%token  bracket_       \[
%token _bracket        \]
// Rest.
%token  colon          :
%token  comma          ,
%token  number         \d+

Règles :

value:
    <true> | <false> | <null> | <number> | string() | object() | array()

string:
    ::quote_:: <string> ::_quote::

#object:
    ::brace_:: pair() ( ::comma:: pair() )* ::_brace::

#pair:
    string() ::colon:: value()

#array:
    ::bracket_:: value() ( ::comma:: value() )* ::_bracket::

Génération d'objets

Objet : aggrégation d'attributs et de méthodes, tous deux annotés par des contrats Praspel

Deux approches pour générer :

  1. appeler le constructeur puis les méthodes pour placer l'objet dans l'état désiré
  2. appeler le constructeur puis définir les attributs de l'objet directement (shunt the encapsulation)

Solution retenue : n°2

Principe

  • générer les paramètres du constructeur à l'aide de sa précondition (@requires)
  • pour chaque attribut, générer une valeur à l'aide de son invariant (@invariant)

Bonus : détecter les références circulaires et réutiliser des objets déjà générés

Synthèse

  • prédicabilité : grâce à instanceof et @invariant
  • générabilité : grâce au nouvel algorithme

3 Génération automatique de tests unitaires

Objectifs de tests

Générer des tests mais… pour tester quoi ?

Quels sont nos objectifs de tests ?

Couverture du code ?

  • boîte noire : le code (corps des méthodes) n'est pas utilisé

Couverture des contrats ?

  • le contrat reflète le comportement des données et du code
  • idée n°1 : couvrir les données
  • idée n°2 : couvrir la structure du contrat
  • hypothèse : couverture du contrat ≈? couverture du code

Critère de couvertures sur les contrats

Critère de Couverture de Clause

  • vise à couvrir la structure d'un contrat

Critère de Couverture de Domaine

  • raffine le Critère de Couverture de Clause
  • vise à couvrir tous les domaines réalistes

Critère de Couverture de Prédicat

  • similaire au Critère de Couverture de Domaine
  • vise à couvrir tous les prédicats (construction \pred(p))

Combinaisons (treillis)

Suite de tests unitaires

  1. choix d'un critère de couverture de contrat
  2. génération d'une suite de tests satisfaisants le critère

Comment l'exécuter ?

  • Praspel sait le faire avec son RAC mais…
  • … l'isolation des tests, les rapports, l'intégration aux multiples outils etc. nécessitent un temps de développement et de maintenance important !

Idée : transformer les suites de tests Praspel en suites écrites avec l'API d'atoum

Pourquoi atoum ?

Les plus :

  • framework de tests de plus en plus utilisé dans l'industrie
  • simple, moderne et intuitif
  • gratuit et open-source
  • intégré ou inclu dans plusieurs IDE et outils majeurs (Netbeans, Eclipse, Vim, Travis-CI, Jenkins, Coveralls…)
  • multiples sortes d'exécutions et formats de rapports
  • hackable

Les moins :

  • pas de générateur automatique de données
  • pas de contrat

var_dump($this->predicate(42, $interval)); // boolean(true)
